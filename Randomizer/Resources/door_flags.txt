================================================================================
 CHIBI-ROBO RANDOMIZER - PER-DOOR OPEN/CLOSED FLAG SYSTEM
================================================================================
 Goal: each room/hallway door has a flag; 0 = closed (default), 1 = open.
 CLOSED must physically block the player (no walking through, no warping).

 A fully-controlled door needs THREE parts, all keyed to its flag:
   1. MODEL   - play the open anim/dir only when open (cosmetic).
   2. WALL    - an invisible collision box spawned with
                `born <id>, 2.d, X, Y, Z, 0.w, exX, exY, exZ, 0.w` (TYPE 2 = wall)
                in the CLOSED branch. The dami objects do NOT block; only a
                born type-2 box does.
   3. MAP-JUMP- early-`return` from the door's map_jump_box .interact handler
                while closed, so the player can't warp to the next room.

 Born world coords = JSON object position * ~98.18 (verified).
 NOTE: every new wall's coords/extents should be eyeballed in Dolphin and
 nudged - the kitchen wall took a couple of tries to sit in the doorway.

 IMPLEMENTATION NOTE (how each DONE door was converted)
 -----------------------------------------------------
 Per door: (a) the load-time init gate was changed from the old story flag to the
 new door flag (so default 0 = closed -> born wall branch runs), and (b) the
 door's map_jump_box handler got an early `return` when the flag is closed.
 Cutscene/ending/reset code that directly opens a door was LEFT ALONE - it opens
 the door in-the-moment (fine) and the door reverts to its flag state on reload.
 New walls (10202/10100/10010) use coords = JSON pos * 98.18 and STILL NEED an
 in-game placement check like the kitchen wall did. (rouka_door_j's wall was moved from
 10200 to 10202 - the original number collided with vanilla's Foyer Teleport pad
 registration, see project_foyer_teleport memory.)

   object worked FUNCTIONALLY (model, open/close, wall) but produced a
   persistent extra empty destination-label box in the Foyer's overhead
   "which door leads where" camera view, tracking the object's live position no
   matter where it was moved, and appearing REGARDLESS of model/data/interact-

   BUILD PIPELINE note below.

 BUILD PIPELINE NOTE - where stage JSON actually comes from
 ------------------------------------------------------------
 `Randomizer/DefaultStages/*.json` is REFERENCE ONLY - not read by the app at
 all; it's just there to see the shape of vanilla stage data.
 `Randomizer/Resources/stageNN.json` is ALSO not the live source for object
 data - Form1.cs's initializeStages() runs `unplug stage export --iso ...` on
 the user's own ISO fresh every time, overwriting the working-directory copy of
 stageNN.json before anything reads it. So the ONLY way to make a persistent
 object-placement change (new door, repositioned object, model swap) is to add
 JObject-mutation code in Form1.cs itself (see the id==24/id==3/id==76 examples
 near the "Edits for Open Upstairs setting" block), NOT to hand-edit a JSON file
 in the repo.
 By contrast, `Randomizer/Resources/stageNN.us` IS the real source for script
 changes - Form1.cs File.Copy's it to stageNN_Edited.us at build time (plus a
 few further string-substitution patches), then assembles that. Editing these
 .us files directly is correct and is how every door/wall/var change in this
 doc was made.

   Foyer<->Basement and Living Room<->Backyard doors, there is no existing
   map_jump_box here at all to repurpose - building this connection would need
   an entirely new trigger object from scratch, not just gating logic. Decided
   NOT to build this (user declined) - documented here so it isn't
   re-investigated from scratch later.
